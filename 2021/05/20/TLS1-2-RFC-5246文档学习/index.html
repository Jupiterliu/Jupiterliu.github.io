文档指定了TLS协议的1.2版本，与TLS协议的1.1版本进行了对比，同时对1.2版本做了详细的介绍。

介绍

TLS协议由两层组成：TLS握手协议TLS记录协议。其具有两个基本属性：连接私有（加密）和可靠（完整性检测-MAC）。

TLS记录协议用于封装各种高层协议，TLS握手协议就是高层协议之一，如下图所示。TLS握手协议允许客户端和服务端在发送第一个字节的数据之前协商好双方的加密算法和密钥等关键要素。TLS握手协议具有的连接安全性：身份验证（公钥算法）、安全的共享密钥协商（无法MITM）和可靠的协商。

TLS协议是独立于应用程序协议的，如下图所示，更高级别的协议可以透明的覆盖在TLS协议之上。

要求的专业术语

在RFC-2119文档中进行查看相关术语，比如“MUST”、“SHALL”、“SHOULD”等等。

与TLSv1.1的不同之处

主要是密码算法协商的改进，对TLSv1.1版本感兴趣的可以看看，这里省略。

目标

TLS协议被应用的目标，4个点，优先级往下依次是：密码安全、互操作性、可拓展性（框架加其他算法和公钥等等）、相对效率。

此文档的目标

该文档主要面向两类人：想要实现该协议的人和进行密码分析的人。因此许多依赖于算法的数据结构和规则在文档中就可以看到。此外该文档不提供服务或则接口定义的相关细节，这也可以说是为了安全着想。

HMAC和伪随机函数

TLS记录层使用带有密钥的MAC（消息认证码）来保证消息的完整性。PRF（伪随机函数）将秘密（secret）、种子（seed）和识别标签（A(i)）作为输入，然后产生任意长度的输出。在本节中，基于HMAC定义一个PRF。

首先定一个数据扩展函数P_hash(secret, data)，公式计算如下，其中HMAC_hash()是散列函数，它拓展一个secret和seed来得到任意长度的输出。其中“+”表示数据之间的串联，A(i)函数的定义是：A(0)=seed, A(i)=HMAC_hash(secret, A(i-1)).

P_-hash(secret, seed) = HMAC_-hash(secret, A(1) + seed) + \HMAC_-hash(secret, A(2) + seed) +\
HMAC_-hash(secret, A(3) + seed) + …

Example: 利用上述函数产生特定长度的数据。如果HMAC_hash()函数是P_SHA256()，一个该函数产生32字节，也就是256比特长度的数据，现在要产生一个80字节长度的数据，则需要用P_SHA256()函数迭代3次到A(3)最终产生96字节长度的数据，这时把最后16字节去除，剩下的80字节数据即为所得数据。

TLS协议的PRF函数通过P_hash加入密钥参数来创建，如下公式所示。其中label参数是ASCII字符串，

PRF(secret, label, seed) = P_-(secret, label + seed)

Example: 字符串中间带有空格的话，也需要讲空格转换成ASCII值，也即转换为0x20的十六进制表达。

TLS记录协议

TLS记录协议传输消息，首先对数据进行分块，然后可选的进行压缩，接下来加上MAC字段，最后加密后进行传输发送。记录协议接收消息就是反过来，解密、验证、解压缩、重组，然后发送给更高级别的客户端。

该文档描述了4种使用记录协议的协议：握手协议（Handshake）、警报协议（Alert）、更改密码规范协议（ChangeCipherSpec）和应用数据协议（Application Data）。

同时为了TLS协议的扩展，记录协议可以支持附加内容，而该内容的规范和值在后面第12节讲述。所以当TLS协议接收到意外的记录类型，将会发送一个“unexpected_message”警报。

连接状态

TLS的连接状态就是TLS记录协议的操作环境，它指定了压缩算法、加密算法和MAC算法，除此之外，这些算法的参数都是已知的。逻辑上来说，现在又4种突出的连接状态：当前读状态和当前写状态、挂起读状态和挂起写状态。所有记录在当前读写状态下进行，通过后面提供的值来设置TLS连接读写状态的安全参数：连接端类型（server或client）、PRF算法、大量数据加密算法、MAC算法、压缩算法、主密钥（48字节）、客户端随机数（32字节）、服务器随机数（32字节）。

TLS记录层通过使用安全的参数来生成下面6个项的值：客户端和服务器的写入MAC值、加密值、向量IV。服务器在接收和处理记录时使用客户端的写入参数，反之亦然。用于生成上面6个项的值的算法在后面6.3节中描述介绍。

每个连接状态包含以下参数：压缩状态、密码状态、MAC密钥、序列号。

密钥计算

通过TLS握手协议得到的相关安全参数以及算法来生成各种密钥。对主密钥进行拓展，可以拓展出客户端/服务器写入MAC密钥、客户端/服务器写入加密密钥。注意有些AEAD加密方式可能会要另外生成客户端/服务器写入IV（向量）。

当密钥和MAC密钥生成之后，主密钥将作为一个墒源。通过下面的计算方式来得到密钥材料，直到得到足够的输出：

key_block = PRF(SecurityParameters.master_secret, “key expansion”,

SecurityParameters.server_random + SecurityParameters.client_random);

然后将上面得到的key_block密钥进行分块，分别得到下面的密钥块：

client_write_MAC_key[SecurityParameters.mac_key_length] server_write_MAC_key[SecurityParameters.mac_key_length] client_write_key[SecurityParameters.enc_key_length] server_write_key[SecurityParameters.enc_key_length] client_write_IV[SecurityParameters.fixed_iv_length] server_write_IV[SecurityParameters.fixed_iv_length]

例子：当前需要数据长度最长的密码组件是AES_256_CBC_SHA256，该密码组件需要2个32字节长度的密钥和2个32字节长度的MAC密钥。

TLS握手协议

TLS握手协议负责协商会话，包括以下组成：会话标识符、对等证书、压缩方式、密码组件及规范、主密钥、会话恢复。

握手协议概述

该协议运行在TLS记录协议之上，协商协议版本、密码组件、加密算法，进行可选的相互验证以及生成共享密钥。

TLS握手协议包括以下步骤：

  1. 交换Hello Message，就算法达成一致，交换随机数，并检查会话恢复；
  2. 交换必要的加密参数，允许客户端和服务器就预主密钥达成一致；
  3. 交换证书和加密信息，允许客户端和服务器进行自身身份验证；
  4. 从预主密钥生成主密钥并交换随机数；
  5. 向TLS记录层提供安全参数；
  6. 允许客户端和服务器验证对等方是否计算了安全正确的参数，以及判断握手是否被篡改。
