国密标准

关注国家商用密码检测中心和国家标准全文公开系统和全国标准信息公共服务平台以及国家密码管理局和中国密码学会和密码行业标准化技术委员会发布的标准。

🌟🌟🌟注意：GMT_0024-2014关于GM的SSL VPN标准已经是2015年的标准，现在2020-11-01已经出台新的标准，也就是GB/T 38636-2020标准（TLCP-传输层密码协议），其中关于密码组件有了新的描述和值，如下图所示：

同时，我国的SM2/3/4/9已经成为了ISO/IEC的标准，以后应用更加广泛

关于商密和RFC的一些标准文档：RFC-8998:ShangMi (SM) Cipher Suites for TLS 1.3

国密SSL/TLS

🌟🌟🌟国密SSL实验室（重点）：好多使用的工具，包括国密Web浏览器、国密Web服务器、国密性能测试、国密工具箱等等。可以从这里面下载Web服务器，搭建并进行TLS-Attacker的测试。一个阿里云的实战博客，该博客实战了GMSSL的Nginx安装和使用。一个相关测试的专栏文章-国密SSL之性能测试。

一个在线工具：SSL/TLS安全评估报告-在线网站检测。其支持国密HTTPS的检测分析。

江南天安Github主页-jntass：提供源码下载和Nginx服务器的安装。

GmSSL：支持国密的密码工具箱，GmSSL作者关志教授的Github主页：GmSSL-guanzhi。其中关志教授将GM-Standards都列了出来，所有国密行业的标准pdf。

大宝CA数字证书测试平台（工具平台/Java/服务器）：该平台有Java国密算法基本组件（JSE），有Java国密SSL算法基础套件，有国密SSL规范的Tomcat组件，有国密SSL规范Nginx服务器。❌（网站未备案，说不定过段时间就可以了）

阿里巴巴的BabaSSL，支持国密操作的OpenSSL开源项目，其代码库。

ATLS：A light TLS implementation used for learning: TLS 1.0 TLS 1.1 TLS 1.2 TLS 1.3 GMSSL 1.1(国密SSL) based on libcrypto.so.

第一次GmSSL环境搭建实验（国密SSL实验室）：

  • 国密SSL协议之Nginx集成：结合前面的国密SSL实验室进行，在CentOS 7环境上实验~~（失败）~~❌❌❌
  1. 先在国密SSL实验室官网下载国密Web服务器：gmssl_openssl_1.1_b4.tar.gz
  2. 在CentOS8虚拟机中，解压文件到/usr/local目录下面，解压后文件为/gmssl
  3. 从Nginx官网下载Nginx：nginx-1.18.0.tar.gz，将conf中的OPENSSL位置进行更改（这里改文件出了一点小差错，还好看出来并解决了）
  4. 从GMSSL中生成证书，然后开启nginx服务
  5. 下载360极速浏览器，访问centos服务器的地址。出错误了，在这里访问CentOS的IP地址并不能成功进行访问。这里能正常访问http但是不能访问https，无论如何都不行，也不知道是不是浏览器的问题。无法解决，换一个环境。

第二次GmSSL环境搭建实验（GmSSL-关志教授）：

  • 搭建国密SSL开发测试环境：结合前面的GmSSL-关志教师-Github进行，在CentOS 7环境上进行实验**（成功）**
  1. 决定换一个，换成guanzhi的GmSSL环境搭建，试一下有无用，如果在不行，就换成TaSSL的环境。
  2. 先安装CentOS 7，接下来安装相关软件，Openssl、Gcc、GmSSL，环境都搭建好了，且都能成功运行测试。
  3. 接下来进行server的创建，然后本地连接服务器，进行国密SSL测试
  4. 出了一个问题，就是提示libcrypto.so.1.1和libssl.so.1.1文件有问题。这里打算重新安装d版本的openssl试一下是否有错误。这次重装了一次GmSSL之后，就没有出现错误了。
  5. 开始打开GmSSL的服务端，试一下国密浏览器是否能连接。
  6. 又出现了一个问题，就是主机上并不能访问虚拟机openssl的ip地址，这就很奇怪了，能相互ping通，但是不能在主机浏览器上进行访问，但是能在虚拟机上对本地进行连接。尝试关闭虚拟机的防火墙之后，主机能对虚拟机进行正常访问。
  7. 通过在Mac上下载GmBrowser并且访问CentOS服务器的地址，成功进行国密连接。
  8. 接下来，使用TLS-Attacker和TLS-Scnner对本地环境进行测试。开始着手TLS-Attacker和TLS-Scanner的熟悉和编程修改优化。

GmSSL实验室的Nginx实验室服务器的支持的密码组件是ECC-SM4-CBC-SM3、ECDHE-SM4-CBC-SM3、ECC-SM4-GCM-SM3、ECDHE-SM4-GCM-SM3，但是并不知道密码组件所代表的值，其中两个是CBC密码模式的组件。是否可以通过Wireshark来抓取服务器的包，然后知道其所代表的值（这估计是不行的）。

正式实验记录（第四次搭建实验）：

遇到的问题：Nginx服务器的密码组件格式是ECDHE-SM4-CBC-SM3形式，而正式标准规定TLS的密码组件格式为ECDHE_SM4_CBC_SM3（下划线分割）格式，GmSSL实验室的服务器是这样，不知道现在其他的服务器是否是如此规定。可以试图修改服务器的组件表达方式。

找到了/usr/local/gmssl/lib目录下的libcrypto.a文件，解压之后发现了sm2、sm3等的obj类型文件，例如sm3.o文件，该文件不可读，打开是乱码，但是百度有方法可以解码后阅读。

  1. 首先测试服务器：https://sm2test.ovssl.cn

    发送ClientHello之后，服务器返回的不是正常的TLSv1.2版本消息，而是SSL版本的Continuation Data消息，一般是大包分块传输，这个服务器可能不好进行测试，毕竟评级是B，挺安全的。

  2. 测试服务器：https://demo.gmssl.cn

    测试发现，无法抓取TLS协议数据包，因为服务器使用的TLS协议版本是GmTLS（0x0101），这个协议版本是Wireshark无法识别，且框架也无法识别的。对国密SSL协议还是不太熟悉，得经过测试修改才能进行正常通信。

  3. 测试服务器：https://ebssec.boc.cn/

    测试之后，可以提前发出ClientHello消息进行通信连接，但是对方服务器并没有进行TLS握手协议消息回复，只能进行TCP的握手和挥手过程，这里很奇怪，可能是服务器协议还存在一些问题。

  4. 测试服务器：https://demo.gmssl.cn:1443

    测试发现无法进行通信，这个1443端口是进行双向国密证书认证测试的端口，可能是因为这个无法建立连接。以后对双证书熟悉且能进行适配之后进行连接测试。同样的，2443端口也不能进行连接测试。

  5. 测试服务器：https://www.globalsign.cn/

    能正常发送TLSv1.2握手协议消息，但是服务器并不进行TLS握手协议消息回复，只能进行基本的TCP握手和挥手。

  6. 测试服务器：https://www.zjca.com.cn

    和第一个测试服务器结果一样，只能接收到Continuation Data的SSL协议消息。

  7. 测试服务器：https://www.trustasia.com

    亚洲诚信服务器可以进行正常通信，可以作为初期测试对象

  8. 测试服务器：https://gm.trustasia.com
